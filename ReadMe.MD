ПОДРОБНЫЙ КОНСПЕКТ ЛЕКЦИИ: Архитектура приложения, слой доступа к данным и требования к проекту

ЧАСТЬ 1. Слой доступа к данным (Data Access Layer) и работа с БД

Преподаватель детально разбирает класс DataAccess, который отвечает за
непосредственное взаимодействие с базой данных (БД) с использованием
технологии ADO.NET.

1. Инициализация и подключение (Connection String)

  - Создание класса: Создается класс DataAccess, который при инициализации
    (через конструктор) должен получать строку подключения (Connection
    String).
  - Проверка строки: Если строка подключения передана, создается объект
    подключения. Если нет (пустая или null) — обязательно выбрасывается
    исключение (Exception), так как работа без строки подключения невозможна.

2. Получение данных (Метод GetTable)

  - Суть: Метод получает данные из источника в виде таблицы (DataTable) по её
    имени.
  - Реализация: Выполняется простейший SQL-запрос вида SELECT * FROM
    [Имя_таблицы].
  - Внутренний вызов: Этот метод под капотом использует базовый метод
    ExecuteQuery (описан ниже).

3. Добавление и обновление данных (InsertRecord, UpdateRecord)

  - Параметры: Методы принимают имя таблицы (TableName) и набор параметров
    (данные для вставки/обновления).
  - Архитектурный совет (НЮАНС!): Преподаватель настоятельно рекомендует не
    передавать параметры поштучно или в виде простых списков. Лучше
    передавать объект, содержащий набор полей (параметров). Если полей
    много, их лучше упаковать в список (List) внутри этого объекта.
  - Обработка параметров: Внутри метода с помощью цикла (например, foreach)
    параметры извлекаются из переданного списка/массива и добавляются в
    SQL-команду.
  - Защита от SQL-инъекций (КРИТИЧНО!): Преподаватель акцентирует внимание на
    маскировке параметров с помощью символа @. Значения не должны вклеиваться
    напрямую в SQL-строку, они должны передаваться через параметры (@paramName),
    чтобы защитить БД от инъекций.
  - Формирование SQL-строки (НЮАНС!): При склеивании строки запроса (имена полей
    через запятую) в конце цикла образуется "лишняя" запятая. Преподаватель
    отмечает, что эту последнюю запятую необходимо программно удалять перед
    выполнением запроса.

4. Базовые методы выполнения запросов

Вся работа с БД строится на двух фундаментальных методах:

Метод ExecuteQuery (Для SELECT запросов)

  - Возвращает: Объект DataTable.
  - Алгоритм работы:
    1.  Создается соединение (SqlConnection).
    2.  Создается команда (SqlCommand), куда передается текст запроса и
        соединение.
    3.  Команда заполняется параметрами.
    4.  Создается адаптер данных (SqlDataAdapter).
    5.  Адаптер заполняет пустой объект DataTable.
    6.  Метод возвращает DataTable.
  - Обработка ошибок: Преподаватель советует оборачивать блок возврата таблицы
    (return table;) в try-catch. Если на этапе формирования или возврата таблицы
    возникает ошибка (вылезает исключение), блок catch должен её перехватить.

Метод ExecuteNonQuery (Для INSERT, UPDATE, DELETE)

  - Возвращает: Целое число (int) — количество измененных, добавленных или
    удаленных строк в БД. По умолчанию возвращает 0.
  - Алгоритм работы: Аналогичен ExecuteQuery (создание соединения, команды,
    параметров), но вместо адаптера вызывается метод
    command.ExecuteNonQuery().

ЧАСТЬ 2. Правила написания кода (Clean Code)

  - Правило "Один метод = Одна операция": Преподаватель категорически против
    создания универсальных "методов-комбайнов", которые делают всё подряд (и
    добавляют, и удаляют, и обновляют в зависимости от переданных флагов).
  - Почему это важно: Слияние логики приводит к избыточному коду и нарушает
    принципы чистого программирования.
  - Решение: Для каждой операции (Insert, Update, Delete) должен быть свой
    отдельный метод. Да, они будут похожи, но они должны быть разделены.

      - Описание архитектуры со схемами (например, клиент-серверная,
        многослойная).
      - Схема потоков данных (Data Flow Diagram).
      - Диаграмма последовательностей (Sequence Diagram) — как компоненты
        общаются между собой во времени.

Глава 3. Реализация и Тестирование

Это глава про то, как вы написали код и проверили его.

  - Реализация архитектуры:
      - Диаграмма классов.
      - Описание используемых интерфейсов (Interfaces).
      - Описание внедрения зависимостей (Dependency Injection).
  - Реализация функционала: Описание работы ключевых алгоритмов и модулей.
  - ТЕСТИРОВАНИЕ (Обязательный раздел, уделено особое внимание!):
      - Модульное тестирование (Unit Tests): Проверка сигнатур методов, проверка
        работы изолированных методов.
      - Мокирование (Mocking): Создание объектов-заглушек ("моков") для
        тестирования модулей в изоляции от реальной базы данных или
        других внешних систем. Это позволяет перенести тестирование на
        "более ранний уровень", не дожидаясь готовности всей системы.
      - Интеграционное тестирование: Проверка того, как связанные модули
        работают вместе.
      - Функциональное (ручное) тестирование: Проверка программы с точки зрения
        конечного пользователя (ввел данные -> нажал кнопку -> получил
        результат).

Заключение

  - Краткий итог: что было сделано, достигнута ли цель, решены ли задачи.

РЕЗЮМЕ ТРЕБОВАНИЙ ПРЕПОДАВАТЕЛЯ (Квинтэссенция)

1.  Безопасность: Обязательная защита от SQL-инъекций (через параметры @).
2.  Архитектура: Строгое разделение ответственности. Один метод делает только
    одну вещь. Никаких универсальных методов-монстров.
3.  Обоснованность: Использование любых фреймворков (EF, DDD) нужно
    аргументировать. Знание базового ADO.NET — обязательно.
4.  Документация: Отчет должен быть структурированным (ТЗ -> Архитектура ->
    Реализация -> Тестирование), без лишней воды. Прототипы интерфейсов не
    должны заменяться скриншотами готовой программы.
5.  Тестирование: Обязательное наличие Unit-тестов и понимание работы с
    Mock-объектами (заглушками).
